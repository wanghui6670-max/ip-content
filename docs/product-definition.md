# IP content product definition

## 项目定位

`ip-content` 的目标不是做一个普通内容看板，也不是单纯同步飞书表。

它要服务的是 **IP 打造流程**：把一个人或一个品牌的外部观察、真实经验、知识沉淀、选题表达、发布反馈，沉淀成可持续复用的内部操作系统。

一句话定义：

> `ip-content` 是从输入资产到知识库，再到选题、内容输出和复盘增长的 IP 打造操作台。

## 当前问题

现有 dashboard 已经开始解决输出留痕：

- 选题是否进入推进
- 内容任务是否排期
- 发布回执是否建立
- 周复盘是否记录
- 资产是否被复用

但 IP 打造真正的上游还没有被系统化保存：

- 外部博主内容为什么值得采集
- 用户问题从哪里来
- 哪些表达结构可借鉴
- 哪些观点已经沉淀为自己的知识
- 哪些知识卡反复支撑了高质量选题
- 哪些输入资产最终带来了发布表现或转化

后续开发需要优先补齐输入链路，而不是只继续加强输出看板。

## 核心闭环

目标闭环如下：

```text
IP 定位
  -> 博主数据采集
  -> 输入资产
  -> 知识库
  -> 洞察 / 方法论
  -> 选题池
  -> 内容任务
  -> 发布回执
  -> 周复盘
  -> 反哺知识库和 IP 定位
```

这个闭环里，知识库是关键中间层。

没有知识库时，链路容易变成：

```text
看到别人的内容 -> 改写成自己的选题
```

这会让系统变成素材搬运工具。

有知识库时，链路应该是：

```text
看到外部信号 -> 提炼用户问题 -> 更新自己的知识资产 -> 生成符合 IP 定位的选题
```

## 领域层级

### 1. IP 档案

IP 档案是最高层判断标准，用来回答这个 IP 为什么存在。

建议沉淀字段：

- IP 名称
- 定位一句话
- 目标人群
- 核心问题
- 核心承诺
- 差异化观点
- 信任来源
- 转化路径
- 表达边界和禁区

所有输入资产、知识卡和选题，都应该能回到 IP 档案进行判断。

### 2. 博主数据采集

当前已有的“博主数据采集”飞书多维表应该作为输入层入口，而不是被替换掉。

它记录的是外部信号：

- 博主账号
- 平台
- 内容链接
- 标题 / 封面 / 开头
- 发布时间
- 点赞 / 收藏 / 评论 / 转发
- 内容形式
- 所属赛道
- 评论区观察
- 采集原因

这里保存的是“看到了什么”和“为什么值得看”，还不是最终选题。

### 3. 输入资产

输入资产是从博主内容、客户对话、个人经历、产品细节、评论区和历史内容中提炼出的可复用原料。

建议资产类型：

- 用户问题
- 痛点表达
- 反常识观点
- 表达结构
- 案例线索
- 转化线索
- 评论洞察
- 产品细节
- 个人经历
- 失败教训

输入资产必须记录来源，避免变成无出处的灵感堆。

### 4. 知识库

知识库是 IP 的认知底座，不是资料夹。

它保存的是已经被筛选、归纳、验证或准备验证的自有认知资产。

建议知识卡类型：

- 观点卡
- 方法论卡
- 案例卡
- 问题卡
- 术语卡
- 流程卡
- 证据卡
- 反例卡
- 产品 / 服务卡

建议字段：

- 知识卡 ID
- 类型
- 标题
- 核心内容
- 适用人群
- 解决问题
- 支撑证据
- 关联输入资产
- 关联选题
- 成熟度：草稿 / 可用 / 验证中 / 核心资产

知识库的作用是把外部信号编译成自己的表达系统。

### 5. 洞察与母题

洞察是从多个输入资产或知识卡中抽象出来的判断。

母题是可以持续拆分成多平台内容的稳定表达方向。

典型链路：

```text
博主内容：10 个 AI 工具推荐
输入资产：用户喜欢收藏工具清单，但真正的问题是不知道怎么落地
知识卡：AI 学习不是工具优先，而是场景和工作流优先
母题：AI 五层结构
选题：别再碎片化学 AI，先建立五层结构
```

### 6. 选题池

选题池不应该直接复制博主标题。

一条合格选题应该回答：

- 来源输入资产是什么
- 来源知识卡是什么
- 我的观点是什么
- 面向哪个目标用户
- 解决什么问题
- 适配哪个平台
- 最终转化动作是什么

建议规则：

- 每条选题至少关联一个输入资产或知识卡
- 每条选题必须写清“我的观点”
- 每条选题必须能回到 IP 定位
- 复盘时要能看到该选题反哺了哪些知识卡

### 7. 输出与复盘

输出层包括当前 dashboard 已经覆盖的内容：

- 内容任务
- 发布回执
- 周复盘
- 资产复用

但复盘不能只看单条内容表现，还要反哺上游：

- 哪类输入资产更值得采集
- 哪类知识卡更容易形成有效选题
- 哪些观点应该升级为核心资产
- 哪些表达结构适合当前 IP
- 哪些转化动作有效

## 建议飞书表结构

短期内可以继续围绕飞书多维表推进，不急着另起复杂系统。

建议逐步形成这些表：

- `IP档案`
- `博主账号`
- `博主内容采集`
- `输入资产`
- `知识库`
- `母题洞察`
- `选题池`
- `内容任务`
- `发布回执`
- `周复盘`

其中最重要的新增边界是：

```text
博主内容采集 -> 输入资产 -> 知识库 -> 选题池
```

`输入资产` 和 `知识库` 之间不能省略。前者保存原料，后者保存自己的认知。

## 仓库模块职责

### `apps/web-dashboard`

定位为 IP 操作台。

当前先展示输出留痕，后续应增加输入链路和知识库视图。

### `data/feishu-mirror`

定位为飞书镜像数据。

它不定义业务本身，只保存可进入仓库的镜像快照和示例配置。

### `packages/content-domain`

定位为 IP 领域规则层。

后续应逐步承接：

- IP 档案实体
- 输入资产实体
- 知识卡实体
- 母题 / 选题实体
- 状态和成熟度规则
- 从输入到选题的关系校验

### `packages/feishu-sync`

定位为飞书镜像契约和同步层。

后续不只校验输出表，也要校验输入链路表是否完整。

### `tools/ip-cli`

定位为本地操作命令。

后续应支持：

- 校验镜像表
- 校验输入资产到知识库的链路
- 校验选题是否有来源
- 输出可读的本地 preflight report

## 下一步优先级

建议后续任务顺序调整为：

1. 确认飞书表模型：先确认“博主内容采集、输入资产、知识库、选题池”的字段和关联。
2. 扩展 `feishu-sync` 表契约：把输入链路表纳入 mirror validation。
3. 扩展 `content-domain` 实体：新增 IP profile、input asset、knowledge card、topic source relation。
4. 增加 dashboard 输入链路视图：先展示输入资产和知识卡，不急着做复杂编辑。
5. 再推进 CLI：把校验和本地报告沉淀成稳定命令。

## 不做什么

短期不做：

- 直接接入复杂在线同步 Worker
- 在 dashboard 内重做完整飞书编辑器
- 直接用 AI 根据博主标题批量生成选题
- 只按发布数据优化内容，而不追溯输入资产和知识卡

这些都容易把项目带回“输出看板”或“素材搬运工具”。
